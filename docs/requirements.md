**図書貸出管理システム 要件定義書**

**文書番号:** RDS-LIB-001

**版数:** 1.0

**作成日:** 202X年X月X日

## **1\. はじめに**

### **1.1 背景と目的**

現在、手作業および旧システムで行われている図書貸出・返却・蔵書管理業務において、業務効率の低下と人為的ミスの発生が課題となっている。本システムは、図書の貸出・返却の自動化、蔵書検索の利便性向上、および在庫管理の適正化を実現し、利用者サービスの向上と図書館運営の効率化を図ることを目的とする。

### **1.2 対象範囲（スコープ）**

本システムは、以下の業務領域を対象とする。

1. **利用者向けサービス（OPAC）**: 蔵書検索、予約、マイページ機能  
2. **カウンター業務**: 貸出、返却、利用者登録、延滞対応  
3. **管理者業務**: 蔵書登録、棚卸（蔵書点検）、統計情報出力  
4. **バッチ処理**: 督促メール送信、予約期限切れ処理

## ---

**2\. 業務フローと全体像**

システムの利用者は大きく「利用者（会員）」と「図書館職員（管理者）」に分類される。

### **2.1 業務概念図**

* **利用者**: Webブラウザまたは館内端末から蔵書を検索し、予約を行う。  
* **職員**: カウンターにてバーコード/RFIDリーダーを使用し、貸出・返却処理を行う。  
* **システム**: 夜間バッチにて貸出期限のチェックを行い、延滞者リストを作成・通知する。

## ---

**3\. 機能要件 (Functional Requirements)**

機能要件を以下のカテゴリに分類し定義する。

### **3.1 利用者向け機能 (Front-end)**

| ID | 機能名称 | 要件詳細 | 備考 |
| :---- | :---- | :---- | :---- |
| **FR-01** | **蔵書検索** | 書名、著者名、ISBN、出版社、カテゴリによる複合検索が可能であること。 検索結果に「貸出可」「貸出中」「予約中」のステータスを表示すること。 | あいまい検索対応 |
| **FR-02** | **貸出予約** | 「貸出中」の図書に対して予約ができること。 1人あたりの最大予約冊数（例: 5冊）を制限すること。 |  |
| **FR-03** | **マイページ** | 現在の貸出状況、返却期限、予約状況、貸出履歴が閲覧できること。 |  |
| **FR-04** | **貸出延長** | 予約が入っていない、かつ延滞していない図書に限り、Web上で1回のみ貸出期間の延長ができること。 |  |

### **3.2 カウンター・管理業務機能 (Back-office)**

| ID | 機能名称 | 要件詳細 | 備考 |
| :---- | :---- | :---- | :---- |
| **FR-11** | **貸出処理** | 会員証と図書バーコードを読み取り、貸出処理を行う。 延滞図書がある場合、または貸出上限冊数（例: 10冊）を超える場合は貸出をブロックする。 | アラート機能必須 |
| **FR-12** | **返却処理** | 図書バーコードを読み取り、返却処理を行う。 返却ポストからの返却（一括処理）にも対応する。 延滞日数を自動計算し、延滞金が発生する場合は画面に表示する。 |  |
| **FR-13** | **会員管理** | 新規会員登録、更新、削除（退会）、会員証発行（バーコード印刷）機能。 有効期限切れ会員の抽出機能。 |  |
| **FR-14** | **蔵書管理** | 新規図書の登録（書誌情報の入力、ISBN書誌情報取得API連携）、廃棄処理、所在情報の更新。 |  |
| **FR-15** | **棚卸（蔵書点検）** | ハンディターミナル等で読み取ったデータとシステム上の在庫データを突合し、不明本リストを出力する。 |  |

### **3.3 バッチ・システム機能**

| ID | 機能名称 | 要件詳細 |
| :---- | :---- | :---- |
| **FR-21** | **督促処理** | 返却期限をX日過ぎた利用者に対し、自動で督促メールを送信する。 |
| **FR-22** | **予約割当** | 返却された図書に予約が入っている場合、ステータスを「取置中」に変更し、予約者にメール通知を行う。 |
| **FR-23** | **統計出力** | 月次・年次の貸出ランキング、カテゴリ別利用統計、利用者属性別統計をCSV/PDFで出力する。 |

## ---

**4\. データ要件 (Data Requirements)**

本システムで管理すべき主要なエンティティと関連性は以下の通りである。

### **4.1 主要データ項目**

* **書誌情報**: ISBN、タイトル、著者、出版社、出版年、カテゴリコード  
* **所蔵情報**: 図書ID（ユニーク）、所蔵館、請求記号、状態（配架、貸出中、汚損、廃棄）  
* **会員情報**: 会員ID、氏名、住所、電話番号、メールアドレス、登録日、最終利用日、ステータス（有効、停止）  
* **貸出履歴**: トランザクションID、会員ID、図書ID、貸出日、返却予定日、返却日  
* **予約情報**: 予約ID、会員ID、ISBN（または図書ID）、予約日、取置期限

## ---

**5\. 非機能要件 (Non-Functional Requirements)**

### **5.1 性能・拡張性 (Performance & Scalability)**

* **レスポンス**: 蔵書検索において、10万件のデータに対し検索結果の表示を **2秒以内** に完了すること。  
* **同時接続数**: ピーク時において、同時アクセス数 **500ユーザ** に耐えうること。  
* **データ量**: 今後10年間の増加を見込み、最大蔵書数 **50万冊**、会員数 **5万人** まで拡張なしで対応可能であること。

### **5.2 可用性 (Availability)**

* **稼働時間**: メンテナンス時間を除き、24時間365日の稼働を保証する（Web OPAC）。  
* **障害対策**: データベースは日次でフルバックアップ、1時間ごとにトランザクションログのバックアップを取得する。

### **5.3 セキュリティ (Security)**

* **個人情報保護**: 会員情報は暗号化してデータベースに保存すること（AES-256準拠）。  
* **アクセス制御**: 図書館職員には権限レベル（一般職員、管理者、システム管理者）を設定し、操作可能な機能を制限すること。  
* **ログ管理**: 全ての貸出・返却操作、および会員情報の閲覧・変更操作は、操作ログとして「いつ・誰が・何を」記録し、3年間保管すること。

### **5.4 インターフェース (Interface)**

* **ハードウェア連携**: 一般的なバーコードリーダー（JAN/Code39対応）およびレシートプリンターとドライバなしで連携可能であること（Webブラウザベース推奨）。  
* **外部連携**: 国立国会図書館API等の外部書誌情報DBと連携し、ISBN入力のみで書誌情報を自動取得できること。

## ---

**6\. 制約条件・前提事項**

* **クライアント環境**: 職員用端末はWindows 10/11、ブラウザはGoogle Chrome / Microsoft Edgeの最新版とする。  
* **ネットワーク**: 職員用端末は閉域網（イントラネット）内からのアクセスを基本とし、外部からの管理画面アクセスはVPN必須とする。  
* **開発期間**: 要件定義完了後、6ヶ月以内のリリースを目標とする。
